# Article Writer — Multi-Agent AI System

A two-agent AI network that autonomously researches a topic, writes a 15-page LaTeX academic article in MDPI journal style, and iteratively reviews and edits it through 3–4 structured review cycles.

## Quick Start

```bash
# 1. Clone and enter the project
cd HW3Version1

# 2. Copy and fill environment variables
cp .env-example .env
# Edit .env — set LLM_PROVIDER and API keys

# 3. Install dependencies
uv sync

# 4. Run the full pipeline (research + write + review + compile)
uv run python src/main.py --mode full

# 5. Research only
uv run python src/main.py --mode research

# 6. Write from existing research.md
uv run python src/main.py --mode write
```

## Environment Variables

| Variable | Required | Description |
|---|---|---|
| `LLM_PROVIDER` | Yes | `anthropic` (Claude Sonnet) or `google` (Gemini 1.5 Pro) |
| `ANTHROPIC_API_KEY` | When `LLM_PROVIDER=anthropic` | console.anthropic.com |
| `GEMINI_API_KEY` | Always (used for web search) | aistudio.google.com |
| `PERPLEXITY_API_KEY` | Optional | Fallback web search |

**Model switching:** Set `LLM_PROVIDER=anthropic` to write with Claude Sonnet 4.6 (default for Nagham's partner); set `LLM_PROVIDER=google` for Gemini 1.5 Pro (Nagham's setup). Search tools always use Gemini + Perplexity regardless of provider.

## How It Works

### Architecture

```
User
 │
 ├─► ResearcherAgent (CrewAI, HITL)
 │       │  Tools: deep_research, researcher_handler, citation_extractor, content_filter
 │       │  Search: Gemini 1.5 Pro → Perplexity fallback
 │       │  Output: data/research.md
 │       └─► Human approves each search batch before next batch
 │
 └─► WriterAgent (CrewAI, deterministic)
         │
         ├─ Phase 1: ContextLoader loads guideline + research + profiles + PDF few-shots
         ├─ Phase 2: DraftGenerator → draft_v1.tex
         └─ Phase 3: ReviewLoop (3–4 iterations)
                  │
                  ├─► Reviewer (SEPARATE CONTEXT)
                  │       Sees: draft + guideline + research + profiles ONLY
                  │       Output: review_v{N}.json  (ArticleReview Pydantic model)
                  │         ArticleReview.comments: list[ReviewComment]
                  │           ReviewComment.profile    — which constraint was violated
                  │           ReviewComment.location   — where in the article
                  │           ReviewComment.comment    — what is wrong (1-2 lines)
                  │         ArticleReview.overall_score — float 0–10
                  │         ArticleReview.pass_fail    — "PASS" | "FAIL"
                  │
                  └─► Editor (FULL CONTEXT)
                          Sees: draft + reviews + guideline + research + profiles + few-shots
                          Applies corrections in priority order:
                            1. Guideline violations (Coverage, Accuracy, Citation)
                            2. Research/accuracy violations
                            3. Profile violations (Structure, Terminology, Characters)
                          Output: draft_v{N+1}.tex

         Final: draft_final.tex → lualatex × 4 passes → article_final.pdf
```

### Agent Contexts (Strictly Enforced)

| Agent | Can See | Cannot See |
|---|---|---|
| **Researcher** | guideline, web search results, user feedback | draft, profiles, few-shots |
| **Writer (Draft + Edit)** | guideline, research, all 3 profiles, few-shot PDFs | reviewer output during drafting |
| **Reviewer** | draft, guideline, research, profiles | few-shot examples, writer system prompt |
| **Editor** | draft, reviewer comments, guideline, research, profiles, few-shots | — (has full context) |

### Tracing and Metrics

Every LLM call and tool invocation is logged:

- **`results/traces.jsonl`** — `{ts, step, provider, model, input_tokens, output_tokens, input, output, metadata}` per call
- **`results/metrics.jsonl`** — `{ts, step, model, latency_ms, input_tokens, output_tokens, total_tokens, cost_usd}` per call

### Eval Dataset Pipeline

```
1. ArticleExtractor  — reads MDPI PDFs (13–20 pages) via PyMuPDF
2. DatasetBuilder    — LLM labels each article PASS/FAIL + 3-sentence critique
                        splits: ~80% train / ~10% dev / ~10% test  (saved as JSONL)
3. ArticleJudge      — LLM judge predicts PASS/FAIL given guideline + research + article
4. JudgeLoop         — runs judge on dev split, computes F1, refines prompt, repeats
                        stops when F1 ≥ 0.80 or max_iterations (default 5) reached
5. Final validation  — runs winning judge on test split (held-out throughout)
```

## Configuration

**`config/setup.json`** — all behaviour is controlled here:
```json
{
  "llm": {
    "provider": "anthropic",
    "anthropic_model": "claude-sonnet-4-6",
    "gemini_model": "gemini-1.5-pro",
    "temperature": 0.3
  },
  "writing": {
    "review_iterations": 3,
    "max_evaluator_iterations": 3,
    "score_threshold": 8.0
  },
  "tracing": {
    "enabled": true,
    "traces_file": "results/traces.jsonl",
    "metrics_file": "results/metrics.jsonl"
  }
}
```
