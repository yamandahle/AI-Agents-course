## Problem Statement

Two distinct bugs degrade the output PDF quality:

### Bug 1 — Raw Citation Keys Shown in Text
33 citation keys generated by the LLM do not exist in `references.bib`. `biblatex`/`biber` cannot resolve them, so they render as literal text like `[russell2010artificial]` or `[nih_ethical_implications]` instead of `[1]`, `[2]`, etc. This is visible in Sections 1 and 1.1 (academic paper keys) and throughout ethical/operational sections (verbose website keys).

**Root cause**: The LLM freely invents citation keys. The first pipeline used short keys (`nih`, `fda`, `oracle`) that matched our seed `references.bib`. The second pipeline generated verbose new keys (`nih_ethical_implications`, `fda_ai_ml_samd`, etc.) and real academic paper keys (`russell2010artificial`, `topol2019deep`, etc.) — none of which exist in the bib file.

**Secondary effect**: When raw English key text (`[nih_ethical_implications]`) sits inside a `\begin{hebrew}...\end{hebrew}` block, the Unicode BiDi algorithm must reconcile RTL Hebrew with LTR Latin text. This produces the "reversed character string artifacts" the user observed — the BiDi algorithm pushes the Latin characters to an unexpected position within the RTL paragraph.

### Bug 2 — Mirror-Image BiDi Layout with No Visual Separation
The BiDi section (Section 5) presents each subsection as a Hebrew paragraph immediately followed by its identical English translation. This creates a "mirror image" layout that:
- Looks like copy-paste duplication rather than purposeful bilingual formatting
- Provides no visual cue that the two blocks are the same content in different languages
- Amplifies the appearance of any BiDi rendering artifacts

## Solution

### Fix 1 — Add all 33 missing BibTeX entries to `references.bib`
- 11 academic papers: `russell2010artificial`, `topol2019deep`, `shah2019artificial`, `lecun2015deep`, `davenport2019ai`, `rajpurkar2017chexnet`, `mckinney2020international`, `campanella2019deep`, `saito2015precision`, `fleming2018artificial`, `wang2019deep`
- 22 website/misc entries: `oracle_cloud`, `nih_ethical_implications`, `nih_patient_outcomes`, `nih_patient_satisfaction`, `televox_patient_engagement`, `deepscribe_ai_medical_scribe`, `jll_commercial_real_estate`, `dataart_digital_transformation`, `practolytics_medical_billing`, `alation_data_catalog`, `ey_global_regulatory`, `fda_ai_ml_samd`, `gov_uk_mhra_roadmap`, `regdesk_eu_ai_act`, `relias_healthcare_lms`, `zynxhealth_clinical_decision_support`, `google_health`, `ibm_watson_health`, `infermedica_platform`, `microsoft_azure_ai`, `pathai`, `tempus`

### Fix 2 — `tcolorbox`-based BiDi block containers
Replace bare `\begin{hebrew}/\begin{english}` pairs with a labeled `tcolorbox` pair:
- Hebrew block: blue-tinted box with "עברית" header (right-side title)
- English block: gray-tinted box with "English Translation" header
- Visual separation via `\medskip` between the two boxes

This makes bilingual content clearly readable, removes the "mirror" perception, and prevents BiDi algorithm from encountering unmarked English within RTL context.

### Fix 3 — Update `_draft_prompt.py` with tcolorbox template + citation guard
- Add `\usepackage{tcolorbox}` and environment definitions to the preamble section
- Replace the bare BiDi block template with the tcolorbox version
- Add a hard rule: **only use citation keys explicitly listed** in the seed list provided in the prompt

## Success Criteria

3. BiDi section shows clearly labeled, visually separated Hebrew/English boxes
4. No "reversed character" artifacts in any paragraph
